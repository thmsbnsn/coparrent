# CoParrent Buyer Demo Script

_Last updated: 2026-03-20_

## Goal

The demo should make one point clearly:

CoParrent is real, broad, and close enough to market that a buyer would save serious time by acquiring it instead of starting from zero.

This is not where every edge case gets proven. This is where the buyer sees that the hard work is already done.

## Demo Length

Target:
- **7-10 minutes live**
- **5-7 minutes recorded**

## Demo Rules

Do:

- move quickly through the product footprint
- emphasize the amount of product thinking already baked in
- show features that are stable and visually persuasive
- use seeded demo data
- be direct about what is done versus what still needs live verification

Do not:

- improvise if the environment is unstable
- demo flows that still depend on unresolved runtime verification
- overclaim GPS, certified court exports, or native mobile apps
- make the buyer wait through setup or account creation

## Recommended Demo Accounts

Prepare these before any buyer call:

- `Parent A` account
- `Co-parent` account
- `Third-party` or `professional` account
- one family with realistic children, schedules, expenses, messages, and documents
- one paid/power scenario enabled

## Suggested Demo Narrative

Use simple framing:

"This is already a real product, not just a front end. The value is that most of the hard category work has already been done."

## Demo Flow

### 1. Start on the dashboard

Open on the dashboard and let the buyer see the breadth right away.

Show:

- dashboard state
- navigation breadth
- signs that this is an operating product, not a thin prototype

Do not linger here. The only goal is to establish scope fast.

### 2. Show the parenting calendar

Show:

- custody calendar
- schedule setup or schedule detail
- exchange defaults and locations
- sports or event integration if the seed data supports it

What to emphasize:

The scheduling model already exists. A buyer is not inventing the calendar product from scratch.

### 3. Show messaging and court-friendly records

Show:

- family or direct thread
- read receipts
- message history
- AI tone assistance if stable in the current environment
- court-record language or exports

What to emphasize:

This is not just chat. Messaging was designed with records and family roles in mind.

Important nuance:

- You may say messages are immutable.
- Do **not** say exports are already certified or affidavit-grade unless that is added later.

### 4. Show child info and documents

Show:

- child profiles
- medical/school/emergency fields
- document library
- categories and organization

What to emphasize:

This is where the product starts to feel operational. The data model is already shaped around how families actually use the app.

### 5. Show expenses and reimbursements

Show:

- expense list
- receipts
- reimbursement workflow
- premium positioning if relevant

What to emphasize:

The expense layer already exists. If a buyer wants payment rails later, they are extending something, not inventing it.

### 6. Show journal / exchange notes

Show:

- journal entries
- exchange-linked private notes
- exchange check-in if seeded

What to emphasize:

The product already captures private, timestamped exchange notes. Be clear that GPS verification is not part of the current implementation.

### 7. Show family access control

Show:

- third-party or alternate-role experience
- restricted navigation
- family or role-based differences if demo data supports it

What to emphasize:

Role differences are built into the product structure, not bolted on at the UI layer.

### 8. Show premium / monetization layer

Show:

- premium gating
- subscription-related UI
- pricing or upgrade prompts if appropriate

What to emphasize:

The monetization layer already exists. That matters to buyers because it shortens the path to launch.

### 9. End with the strategic takeaway

Close with something plain:

"The reason to buy this is not that every feature is done. The reason to buy it is that the hard product and architecture work has already been paid for."

## Optional Add-Ons

If the environment is ready, optionally show:

- law library
- kids hub
- sports hub
- AI-generated activity flow
- coloring-page generation

Only show these if they are stable in the demo environment.

## Claims You Can Safely Make

- the repo has a passing local verification path
- the product spans the main co-parenting workflow surface
- messaging records are immutable
- AI providers are standardized on OpenRouter in the repo
- the codebase is documented
- the platform is suitable for strategic acquisition and rebrand

## Claims You Should Avoid Unless Re-Verified

- that the public deployment exactly matches the latest repo
- that all runtime flows are fully production-verified
- that exchange check-ins use GPS
- that records are certified or notarized
- that native mobile apps exist

- [ ] target environment is stable
- [ ] screenshots are ready as fallback
- [ ] buyer questions are anticipated
- [ ] current open issues are known before the call
